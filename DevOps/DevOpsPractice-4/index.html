DevOps实践(4) —— DevOps实施的第一步

发表于 2020-03-24 | 分类于 DevOps | 阅读次数:
字数统计: 926 | 阅读时长 ≈ 3

《DevOps实践指南》中提出了DevOps三步法，这三步分别是流动、反馈、持续学习和试验。这个三步法概括了一条比较通用的实现路径。

这里我们主要看看第一步 ———— 流动。

在整个交付过程中，流动的是什么？需求？产品？其实流动的是价值。在这个系列的第一篇博客中就提到过，DevOps要解决的问题是“怎样建设能够快速交付价值并且能够持续改进的现代化的IT组织。”

那么，什么是价值呢？对一个组织来说，生存和发展是第一要务，因此，价值就是能够让企业生存发展的核心资源。对于IT企业来说，要详细分析企业的业务流中哪些带来了价值，哪些没有，可以借鉴传统制造业中的精益思想，使用价值流图（VSM Value Stream Mapping）来进行分析。使用VSM分析一个产品交付所经历的所有的业务流、信息流等。

对于IT组织，VSM就是要说清楚从得到需求开始，怎么一步步的进行设计、开发、测试、部署等工作，最终将可使用的产品交付给用户的过程。通过观察整个流程中的各个环节，找出不合理的、浪费的环境，提升效率。

前面我们提出的通用技巧中也说过，我们要“寻找团队痛点，也就是当前最影响团队效率的事情，或者改进后可以产生最大效益的事情”。我们使用VSM，就是要更准确的找出这个痛点。

在软件企业中，由于职责的划分，每个人关注的都是自己眼前的事情，大家都说不清整个交付过程的方方面面。使用VSM对软件交付过程进行描述，让整个过程可视化，识别出各个环节的依赖关系。

VSM中有几个关键的要素和概念：

  1. 前置时间：Lead Time，简称LT。在DevOps中，指的是需求从提出（创建了需求任务）的时间点开始，一直到最终上线交付给用户为止的时间周期。这个指标体现了软件开发团队的交付速度。DevOps的核心要求之一就是减少这段时长。
  2. 增值活动时间和不增值活动时间（VAT/NVAT Value Added Time/Non-Value Added Time): 我们需要最大化VAT，最小化NVAT，因为根据精益思想，NVAT就是浪费。软件开发行业中有很多典型的NVAT，比如bug修复、需求的变更等等。
  3. 完成度和准确度(%C/A %Complete/Accurate):这个指标用于表明工作的质量，也就是由多少工作因为质量不符合要求而被下游打回要求返工。

前置时间又可以细分为需求前置时间和开发前置时间。

需求前置时间：从需求提出到最终验收通过的时间周期。考察的是团队整体的交付能力。是用户核心感知的周期。

开发前置时间：从需求开始开发，到完成开发、测试、上线，最终验收通过的时间周期。

这个时间的收集，需要依赖人员和平台的配合。平台需要有收集和统计的能力，人员则需要按照规范，及时修改各个任务的状态。

初期，我们主要关注于前置时间和不增值活动时间，并且不增值活动时间也主要关注等待时间。
