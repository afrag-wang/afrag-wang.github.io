DevOps实践指南(3) —— 实施DevOps的两条路

发表于 2020-03-22 | 分类于 DevOps | 阅读次数:
字数统计: 817 | 阅读时长 ≈ 3

在组织内部实施DevOps，通常也称为DevOps转型。所谓转型，实际上就是一种变革。自古以来，变革就有两条路：自顶向下和自底向上。

3.1 自底向上

由一个小部门或者小团队，为了解决团队内部，以及上下游团队的交互中的问题，尝试使用DevOps。

团队较小，那么比较容易达成共识，相关的资源调动相对简单，比较容易在局部取得效果。但是DevOps强调的是打破团队间的隔阂，因此还需要对团队间的合作进行变革。这个时候，应该首先在自己团队内部推行，然后在自己团队所负责的业务有强依赖关系的上下游团队建立联系，逐渐模糊团队之间的边界，构建一个DevOps的共同体。

但是，在团队之间的推进是比较困难的，因此需要取得“高层”的支持。我们先要在自己的团队中做出成绩，并以此为支点，取得“高层”的支持，让我们向上下游团队的推进的阻力最小。要取得“高层”的支持，需要用数据说话。因此，我们需要明确的改进目标，以及对该目标的度量。

3.2 自顶向下

在这种模式下，是企业高层基于自己的判断和意愿，以行政命令的方式，投入企业资源，主动推动DevOps转型。

这种方式下，要提防“上有政策、下有对策”的情况发生，因此需要有明确的目标和可度量的指标。

3.3 通用技巧

无论是走哪条路，要想成功都需要上下一心。要有管理层的支持，还要有来自底层员工的理解和执行。另外，还需要持续的资源投入，才能够最终达成共识并快速落地。

因此，采用这两种方式，都可以采用下面的通用路径：

  1. 选择合适的项目开始，这个项目最好具备以下的特征：
    • 贴近核心业务
    • 倾向敏捷业务
    • 有改进意愿的优先
  2. 寻找团队痛点，也就是当前最影响团队效率的事情，或者改进后可以产生最大效益的事情。
  3. 最开始不要把面铺开的太宽。一是资源有限，好刚要用到刀刃上。另外，要快速的让团队成员和领导层看到效果，坚定信心。最好是一个改进点，一个目标。
  4. 快速展示和持续改进：及时向领导层汇报成果，在团队内部总结和讨论，根据反馈进行优化和改进。

3.4 坚定不移

无论走哪条路，变革都是不容易的。前途是光明的，道路是曲折的。在实施DevOps转型的时候，有时候还会出现效能下降的情况，最典型的是DevOps转型的J型曲线：
